# Plan 17-03 — applyAppearance, CMS colorScheme, API & admin

**Requirements:** DSM-04 (CMS hydration + setTheme), DSM-05, remaining alignment for DSM-01/07 via data defaults  
**Prereq:** 17-01, 17-02  
**Depends on CONTEXT:** D-17-05..08, D-17-09..12, D-17-23, D-17-24

## Goal

Persist and edit `appearance.colorScheme`; extract CMS→CSS variable mapping into `src/theme/applyAppearance.ts`; thin `ThemeSynchronizer` calls `applyAppearance` and synchronizes `next-themes` `setTheme` per hybrid precedence; server merges default `colorScheme: 'system'` when missing in stored JSON.

## Tasks

1. **Types:** Extend `SiteAppearance` in `src/lib/websiteData.ts` with `colorScheme?: 'light' | 'dark' | 'system'`; default in factory/seed object to `'system'`.
2. **Server read path:** In `contentRoutes` (or single content assembler), when parsing `appearance` JSON, **`defaultsDeep`** or manual merge: if `colorScheme` missing, set `'system'` before response.
3. **`src/theme/applyAppearance.ts`:** Move logic from `ThemeSynchronizer` (colors, fonts, radius, shadow, customCss injection) into pure function `applyAppearance(appearance, settings, root?)`. Keep `darkenColor` for `--color-accent2` (D-17-10). Document which vars are brand vs semantic.
4. **`ThemeSynchronizer`:** Call `applyAppearance`; then map CMS `colorScheme` to `setTheme()`:
   - `light` | `dark` | `system` → `setTheme(cmsValue)`
   - Implement D-17-06 (force dark) via `setTheme('dark')` when CMS says dark.
   - When CMS is `light` or `dark`, ensure visitor storage does not win (use `forcedTheme` **or** `enableSystem={cms === 'system'}` pattern — pick one approach and document; `next-themes` v0.4 supports `forcedTheme` when needed).
5. **Admin:** `DesignSystemManager` — add control for `colorScheme` (radio or select), bind to preview + save payload.
6. **Admin preview (D-17-07):** Ensure preview iframe or preview root does not read public visitor `localStorage` — use isolated storage key for preview **or** `disableTransitionOnChange` + `storageKey` scoped to admin only when in `/admin` (decide minimal diff; document in CONTEXT follow-up if needed).
7. **Tests:** Vitest for default merge of `colorScheme` on parsed appearance (if pure fn exported from server util, test there; else shallow client test).
